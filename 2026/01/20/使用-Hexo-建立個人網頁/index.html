1. Hexo 簡介

Hexo 是一個簡單、快速且功能強大的靜態網站產生工具，常用於搭配 GitHub Pages 建立個人網頁。
它支援 Markdown 語法，並擁有大量成熟的主題與外掛生態。

  • 官方網站：http://hexo.io
  • GitHub：https://github.com/hexojs/hexo

2. 工作原理說明

GitHub Pages 只能託管靜態檔案，而個人網頁中常包含文章列表、分類、標籤、分頁等結構化內容。

若每次新增內容都需手動維護頁面與連結，實際上難以長期維護。

Hexo 的運作方式為：

  • 所有內容以 Markdown 檔案形式儲存在本機
  • 透過指令一次性產生完整的靜態頁面（HTML）
  • 將產生後的靜態檔案部署至 GitHub Pages

換言之：

  • 內容撰寫在本機完成
  • 網頁由 Hexo 自動產生
  • GitHub 僅負責提供靜態頁面託管

3. 注意事項

在安裝與使用 Hexo 前，請留意以下事項：

  • 多數指令可於 Windows 的 cmd 或 Git Bash 中執行
    為避免環境差異造成問題，建議統一使用 Git Bash
  • Hexo 不同版本之間差異較大，網路上不少教學基於舊版（如 2.x），需留意避免混用
  • Hexo 專案中存在兩個 _config.yml：
    • 專案根目錄下的全域設定檔
    • 主題資料夾中的主題設定檔

修改設定時需確認實際修改的檔案位置。

6. 產生靜態檔案與本地預覽

在專案目錄中執行以下指令：

hexo g   # 產生靜態檔案
hexo s   # 啟動本地預覽服務

說明：

  • hexo g（generate）
    會在 public/ 資料夾中產生完整的靜態 HTML 檔案
    這些檔案就是未來要部署到 GitHub Pages 的內容

  • hexo s（server）
    啟動本地預覽伺服器，方便在部署前檢視實際畫面

啟動成功後，終端機會顯示類似訊息：

INFO  Hexo is running at http://localhost:4000/

在瀏覽器開啟以下網址即可查看個人網頁：

http://localhost:4000

7. 關於本地預覽無法開啟的常見問題

若瀏覽器一直顯示載入中，但畫面無法顯示，通常原因為：

  • 4000 連接埠已被其他程式佔用

此為 Windows 上常見狀況，並非 Hexo 錯誤。

可透過更換連接埠或釋放佔用程式來解決，相關處理方式可參考：

http://blog.liuxianan.com/windows-port-bind.html

8. 預設內容說明

第一次初始化 Hexo 時，系統已自動建立一篇範例文章：

Hello World

此文章用於展示：

  • Markdown 轉換結果
  • 預設主題的版型與樣式
  • 文章頁與首頁列表效果

後續可直接刪除或修改該文章，替換為自己的內容。

9. 問題排除（清理後重新產生）

如果在操作過程中出現一些難以判斷的異常問題，可先執行清理指令：

hexo clean

此指令會清除 public/ 資料夾內已產生的檔案，
之後再重新產生與預覽：

hexo g
hexo s

10. 上傳之前的重要提醒

在將個人網頁部署到 GitHub 之前，務必先備份原本 GitHub 儲存庫中的所有檔案。

原因說明：

  • Hexo 部署時，會以產生後的靜態檔案為主
  • 若設定不當，可能會覆蓋原本儲存庫中的其他內容
  • 雖然 GitHub 具有版本管理，但自行備份仍是較安全的做法

11. 部署到 GitHub Pages

當所有設定完成後，部署流程本身非常單純：

hexo d

但前提是以下條件必須正確完成。

11.1 SSH Key 設定完成

  • 必須已正確設定 SSH Key
  • 並確認可使用 SSH 連線至 GitHub

11.2 設定 _config.yml 的 deploy 區段

請編輯專案根目錄下的 _config.yml，設定如下：

正確寫法（目前適用）：

deploy:
  type: git
  repository: git@github.com:username/username.github.io.git
  branch: master

branch是master還是main根據你安裝git時的選擇

若直接執行 hexo d 出現以下錯誤訊息：

  • Deployer not found: github
  • Deployer not found: git

代表尚未安裝部署外掛。

建議操作順序

為避免在實作過程中混淆 GitHub 身分、SSH 設定與 Hexo 部署流程，
建議依照以下順序完成各階段設定：

  1. 產生並設定 GitHub SSH Key（ed25519）
    建立本機與 GitHub 之間的安全連線，
    作為後續所有 Git 操作的基礎。

  2. GitHub SSH 設定：一台電腦複數 GitHub 帳號的 SSH 與 Git 身分設定說明
    說明 SSH 與 Git user.name / user.email 的角色區分，
    並建立多帳號環境下穩定、不混帳號的實務設定方式。

  3. 初始化 Hexo：使用 Hexo 建立個人網頁
    建立 Hexo 專案、產生靜態檔案，
    並於本地端完成預覽與基本設定。

  4. 部署至雲端：Hexo 使用 SSH 成功部署至 GitHub Pages（多帳號情境）
    透過已設定完成的 SSH 環境，
    將 Hexo 產生的靜態網站部署至對應的 GitHub Pages 儲存庫。

11.3 安裝部署外掛

請執行：

npm install hexo-deployer-git --save

注意事項：

  • 部署指令務必使用 Git Bash 執行
  • 若使用 cmd，可能會出現：
    Permission denied (publickey)

安裝完成後，再次執行：

hexo d

Hexo 只會提交有異動的檔案，未變更的內容不會重複上傳。

12. 保留 CNAME、README.md 等非文章檔案

部署完成後，若發現 GitHub Pages 上原本的其他檔案消失，請勿慌張。

處理原則如下：

  • 將 非 Markdown 檔案（例如 CNAME、圖片、其他靜態資源）
    放入 source/ 資料夾
  • 這些檔案會在產生時原樣複製到 public/ 目錄

注意事項：

  • Hexo 會將所有 .md 檔轉為 HTML
  • README.md 會被轉成 README.html

因此若需要保留 README.md：

  • 每次產生後，手動將 README.md 複製到 public/
  • 並刪除 README.html

13. 常用 Hexo 指令整理

常見指令

hexo new "postName"          # 新增文章
hexo new page "pageName"    # 新增頁面
hexo generate               # 產生靜態檔案至 public/
hexo server                 # 啟動本地預覽（預設 4000）
hexo deploy                 # 部署到 GitHub Pages
hexo help                   # 查看說明
hexo version                # 查看 Hexo 版本

縮寫指令

hexo n == hexo new
hexo g == hexo generate
hexo s == hexo server
hexo d == hexo deploy

組合指令

hexo s -g   # 產生並本地預覽
hexo d -g   # 產生並部署

14. _config.yml 說明

_config.yml 為 Hexo 的全域設定檔。

注意事項：

  • 每個冒號 : 後面 必須保留一個空格
  • 少空格可能導致設定無法正確解析

15. 撰寫個人網頁內容（文章與頁面）

在 Hexo 專案根目錄執行：

hexo new "my-first-page"

Hexo 會在 _posts/ 資料夾中建立對應的 Markdown 檔案。

範例文章結構如下：

---
title: 文章標題
date: 2026-01-20 10:00:00
categories: 分類名稱
tags: [tag1, tag2]
description: 文章摘要（建議 140 字內）
---

以下為正文內容

使用此指令的好處在於：

  • 會自動產生時間
  • 檔案結構正確

16. 文章與頁面的差異

新增頁面指令：

hexo new page "about"

產生結果：

  • 檔案位置：
    source/about/index.md
  • 最終生成：
    public/about/index.html

差異說明：

  • page 不會出現在文章列表中
  • 適合用於「關於我」、「作品集」、「聯絡方式」等固定頁面
