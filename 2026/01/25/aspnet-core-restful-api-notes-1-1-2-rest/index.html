1.1.2 什麼是 REST

REST，全名為 REpresentational State Transfer，中文常翻為「表述性狀態傳遞」，是一種用來設計 Web 服務的架構風格。

REST 並不是一種技術、框架或語言，而是一組設計原則，用來指導如何設計可擴展、易維護的 Web API。
這個概念最早由 Roy Fielding 在西元 2000 年的博士論文中提出。

REST 是為了解決什麼問題？

在 Web 服務規模逐漸擴大後，系統往往會面臨以下問題：

  • API 結構混亂、難以理解
  • 不同服務之間耦合度過高
  • 擴充新功能時影響既有系統

REST 的目的，是降低系統複雜度，並讓 Web 服務能夠更容易擴展與維護。

資源（Resource）是 REST 的核心

在 REST 中，資源（Resource）是最基本的概念。

任何「可以被命名、被操作的對象」都可以視為一種資源，例如：

  • user
  • order
  • product
  • document

使用 HTTP 方法操作資源

REST 不在 URI 中表達動作，而是透過 HTTP 方法 來描述對資源的操作。

例如：

  • GET：取得指定資源
  • PUT：更新指定資源

表述性狀態傳遞（REST 的意義）

當用戶端對資源發送請求，並透過 HTTP 方法改變資源狀態後，
伺服器會回傳該資源的「表述（Representation）」給請求方。

這個「資源狀態的表述在客戶端與伺服器之間傳遞」的過程，
正是 REpresentational State Transfer 這個名稱的由來。

REST 與 HTTP 的關係

REST 本身並不限定一定要使用 HTTP，
但 HTTP 協議完整滿足 REST 所定義的約束條件，因此成為 REST 最常見的實作方式。

HTTP 所提供的功能，例如：

  • HTTP 方法（GET、POST、PUT、DELETE）
  • 狀態碼（200、404、500）
  • Header 與內容協商

都能自然地對應到 REST 的設計原則，使 REST 能夠以鬆耦合的方式建構 Web 服務。

小結：什麼是 RESTful API？

在實際開發工作中，RESTful API 已經成為前後端溝通的共同語言，就像 HTTP、JSON、Git 一樣。

當工作中提到「熟悉 RESTful API」，通常代表具備以下幾項能力：

一、理解 API 的基本結構與語意

  • 使用 URI 表示資源
  • 透過 HTTP Method 表達操作行為
  • 回傳 JSON 資料格式
  • 能正確判讀常見 HTTP 狀態碼

二、能完成 API 的實務整合

  • 前端使用 fetch 或 axios
  • Blazor 使用 HttpClient
  • MVC 呼叫後端 API
  • 能處理授權資訊與錯誤狀態

三、遵循基本 REST 設計原則

  • 避免將動作寫入 URI
  • 採用資源導向設計
  • 搭配 HTTP Method，維持 API 結構清楚且一致

總結來說，RESTful API 是一種溝通規範，只要能看得懂、串得起，並遵循基本 REST 設計，即符合實務上的基本要求。

本節僅介紹 REST 的基本概念與核心思想。
接下來將進一步說明 RESTful API 如何實際應用在 Web API 的設計中。
